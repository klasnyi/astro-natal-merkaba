# Astro Natal МерКаБа — скил построения натальной карты

## Когда активироваться

Активируйся когда пользователь:
- просит "натальную карту", "астрологический расчёт", "астрологический разбор"
- пишет "построй карту", "посчитай астрологию", "натал по дате рождения"
- упоминает "ведический гороскоп", "западная астрология", "накшатры", "асцендент"

## Что делает скил

Рассчитывает натальную астрологическую карту и генерирует стилизованный DOCX-отчёт.

**Технологии:** Swiss Ephemeris через библиотеку kerykeion (< 0.001° точность), python-docx для DOCX, matplotlib для PNG-колеса карты.

**Поддерживаемые системы:**
- **Западная** — тропический зодиак, система домов Плацидус
- **Ведическая** — сидерический зодиак (аянамша Лахири), 27 накшатр, система домов Whole Sign

## Поток работы

### Шаг 1 — Запрос данных (chat-native, НЕ через AskUserQuestion)

Когда пользователь активирует скил, напиши обычное текстовое сообщение:

> Для натальной карты нужно 5 данных:
>
> 1. **Дата рождения** (ДД.ММ.ГГГГ) — например 13.08.1993
> 2. **Время рождения** (ЧЧ:ММ по местному времени) — или скажи "время неизвестно"
> 3. **Город рождения** (город + страна если неочевидно) — например "Москва" или "Алматы, Казахстан"
> 4. **Имя** — как подписать отчёт (только имя, или имя+фамилия)
> 5. **Система**: западная (Плацидус, тропический) или ведическая (Лахири, накшатры) — или "обе"
>
> Можешь прислать одним сообщением.

**КРИТИЧНО: НЕ используй AskUserQuestion с опциями для этих полей.** Все значения — свободный ввод, не закрытый список. Жди текстовый ответ пользователя и парси его.

### Шаг 2 — Парсинг ответа

Извлеки из ответа:
- **дата** — ищи ДД.ММ.ГГГГ (допускай разделители `.` `/` `-`, допускай одну цифру в дне/месяце)
- **время** — ищи ЧЧ:ММ; если написано "не знаю", "неизвестно", "nope", "нет" — `has_time=false`
- **город** — остальное что похоже на топоним (любой язык)
- **имя** — одно-три слова кириллицы, скорее всего после двоеточия или в контексте
- **система** — ищи слова: "западн" → western; "ведич" или "лахири" или "накшатр" → vedic; "обе" → оба прогона

Если что-то непонятно — переспроси по конкретному полю (обычным текстом, не AskUserQuestion).

### Шаг 3 — Расчёт (build_chart.py)

Запусти:

```bash
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/build_chart.py \
  --date ДД.ММ.ГГГГ \
  [--time ЧЧ:ММ] \
  --city "ГОРОД" \
  --name "ИМЯ" \
  --system western|vedic \
  --outdir /tmp/astro-<name>-<date>
```

Если пользователь хочет **обе** системы — запусти дважды с разным `--system` и разным `--outdir`.

На выход получишь:
- `<name>_<date>.json` — данные карты (планеты, дома, аспекты, распределения)
- `<name>_<date>_wheel.png` — колесо карты (встроится в DOCX)

**Принципы интерпретации — КРИТИЧНЫ для качества:**

1. **Конкретика, а не общие слова.** Плохо: "у вас сильное Солнце". Хорошо: "Солнце во Льве 20.32° в 1 доме в 1.5° от вашего Асцендента, в соединении с Хироном в 27° — это означает, что ваше ядро идентичности буквально слито с тем, как вы появляетесь в мире, и с вашей глубокой раной целительства".

2. **Привязка к градусам.** Анаретический градус (29°), 0° и средние 15° — специфичны. Упоминай их, если они значимы в карте.

3. **Диспозиторы (управители знаков).** Например Солнце в Рыбах управляется Нептуном — смотри где Нептун, это усилитель. Для каждой важной планеты упоминай, где её диспозитор, если это меняет смысл.

4. **Синтез, не список.** Плохо — отдельно описывать Солнце и отдельно Хирон, когда они соединены. Хорошо — описывать связку как одно целое.

5. **Специфика знака в доме.** Лев в 1 доме ≠ Лев в 7 доме ≠ Лев в 10 доме. Проявление разное.

6. **Аспекты — центральны.** 3 самых точных аспекта (орб < 1°) — это сердце карты. Уделяй им 2-3 абзаца каждому, с практическими проявлениями.

7. **Баланс света и тени.** У каждого архетипа — дар и вызов. Не льсти, не пугай. Покажи задачу.

8. **Архетипы, а не события.** Не "вы разведётесь в 30" (это не ваше дело), а "ваш Сатурн в 7 доме показывает долгую школу партнёрства".

9. **Жизненные примеры.** "В работе это проявляется как X, в отношениях — как Y, в семье — как Z". Делать карту осязаемой.

10. **Русский язык.** Обращение на "вы", литературно, но не пышно. Избегай эзотерического жаргона без объяснений.

11. **Объём.** Итоговый DOCX должен быть **12-25 страниц**. Не 5-6. Если получается мало — копай глубже.

12. **Личные данные.** Не упоминай пользователя по имени-фамилии в теле интерпретаций — только обращение "вы". Имя использует только шапка DOCX из chart.json.

### Шаг 5 — Рендер DOCX (render_docx.py)

## Кэш натальных карт

После расчёта `build_chart.py` **автоматически сохраняет** карту в:

```
~/.astro-natal-merkaba/cache/<name_slug>_<YYYY-MM-DD>_<city_slug>_<system>.json
```

Например: `~/.astro-natal-merkaba/cache/dmitriy_1993-08-13_moskva_western.json`.

Это нужно чтобы расширенные техники (транзиты, прогрессии, синастрия, ректификация и т.д.) могли работать поверх готового натала **без повторного запроса данных у пользователя**.

### Команды для управления кэшем

```bash
# Список всех сохранённых карт
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/astro_helpers.py list

# Поиск по имени (нечёткий)
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/astro_helpers.py find Дмитрий

# Путь к директории кэша
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/astro_helpers.py path
```

## Расчёт для другого человека (client-mode)

Скил поддерживает работу с несколькими людьми (для астрологов, консультантов, для семьи).

**Если пользователь просит карту другого человека** ("построй карту для Ани", "посчитай Маше Ивановой"):

1. **Извлеки имя другого человека** из запроса (используй его в `--name`).
2. **Запроси данные именно по этому человеку** (дата/время/город — те же 5 полей).
3. **Сохрани в кэш** под именем этого человека (это происходит автоматически в `build_chart.py`).
4. **Если в кэше уже есть карты** разных людей — НЕ путай их. Каждая карта идентифицируется по slug `<имя>_<дата>_<город>_<система>`.

**Если пользователь просит расширенную технику для конкретного человека** ("транзиты Ани сейчас", "соляр Маше на этот год"):

1. **Сначала проверь кэш** командой `astro_helpers.py find <имя>`.
2. Если карта найдена — используй её JSON для расширения.
3. Если не найдена — сначала запроси данные и построй натал, потом запусти расширение.

**Конфиденциальность:**
- Кэш — локальный, на машине пользователя. Не шарится между Cowork-сессиями.
- В отчётах не упоминай имена других сохранённых клиентов.
- Если пользователь просит «удалить карту X» — удали соответствующий файл из `~/.astro-natal-merkaba/cache/`.

## Расширение: Транзиты ✅

Транзиты показывают, какие текущие движения планет активируют натальную карту прямо сейчас и в ближайшее время.

### Триггеры для активации транзитного режима

- "транзиты", "мои транзиты", "транзиты на сегодня"
- "что сейчас активирует мою карту"
- "что меня ждёт", "прогноз на месяц", "прогноз на ближайшее"
- "транзиты для X" (где X — другой человек)
- "что сейчас в небе для меня"

### Поток работы (transits)

**Шаг T1 — Идентификация субъекта.**

Если пользователь не указал имя — спроси: «Транзиты для кого? Для тебя или для другого человека?». Если для другого — спроси имя.

**Шаг T2 — Поиск натала в кэше.**

```bash
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/astro_helpers.py find <имя>
```

- Если карта найдена — используй её путь.
- Если не найдена — запроси данные (Шаги 1-3 натального флоу) и собери натал. Он автоматически попадёт в кэш — затем продолжай с транзитом.

**Шаг T3 — Запуск расчёта.**

```bash
python3.11 ~/.claude/skills/astro-natal-merkaba/scripts/build_transits.py \
  --natal <путь_к_кэшу_натала> \
```

## Edge cases и как их обработать

| Ситуация | Действие |
|---|---|
| Пользователь не знает время | `has_time=false`, карта без асцендента/домов, только Солнце/Луна/планеты в знаках. В отчёте пропускаются секции "Дома". Секция про Асцендент заменяется пометкой "Асцендент требует точного времени рождения." |
| Ректификация (уточнение времени по событиям) | **Запланирована как Фаза 11** (модуль `build_rectification.py`). Зависит от готовых модулей транзитов и прогрессий. Пока модуль не готов — мягко скажи пользователю: "Ректификация в разработке. Можем построить солнечную карту по дате (без асцендента и домов), а позже, когда время уточнится — пересчитать полную карту." |
| Город не найден геокодером | Попросить уточнить (например добавить страну). Если всё равно не находится — попросить координаты (широта, долгота). |
| Тайм-зона в дате-времени амбигуозна (переход на летнее время, до 1981 в СССР) | Использовать локальную tz города через timezonefinder. Если пользователь уверен в UTC — можно добавить флаг `--tz UTC` явно. |
| Пользователь просит "обе системы" | Построить дважды: `--system western` + `--system vedic`. Два JSON, два PNG, два DOCX. В итоге можно сделать ОДИН DOCX с двумя разделами, но по-умолчанию — два отдельных файла. |
| Запрос типа "а что если время другое" | Можно пересчитать карту на разное время — это важная астрологическая техника (см. ректификация). Показать дифф (см. пример сравнения 4:30/5:30). |

## Как НЕ делать

- **Не используй AskUserQuestion** для ввода даты/времени/города/имени — это свободный ввод, а не выбор из списка.
- **Не пиши интерпретации в reference JSON**, они должны оставаться справочниками (архетипы, ключевые слова).
- **Не храни персональные данные пользователей** в папке скила — все выходы в `/tmp/astro-*`.
- **Не делай "предсказаний"** в стиле "вы станете врачом". Говори о паттернах и тенденциях.
- **Не упоминай автора скила в интерпретациях** — атрибуция (имя автора + контакт) указывается только в колофоне DOCX.
